# Inside the Extensy Pipeline

To build effectively with Extensy, it helps to understand what's happening behind the scenes. Here's a look at the architecture of your generated extensions.

## The Generation Pipeline

Behind the scenes, Extensy is powered by the **Sidekick Engine**, an autonomous workflow built on top of LangGraph. It orchestrates multiple specialized AI nodes to automatically generate, style, test, and package your extension.

1. **Initial Router**: Determines the generation path based on your subscription tier.
2. **Architect Node (Pro/Max)**: Extracts a structured blueprint (permissions, features) from your raw prompt using advanced models like `claude-sonnet-4-5`.
3. **Researcher Node (Max)**: Augments context by retrieving high-quality Chrome MV3 patterns from the Nia API.
4. **Coder Node**: The heavy lifter. It writes the `manifest.json`, background workers, content scripts, and core logic.
5. **UI Designer Node (Pro/Max)**: Refines the generated HTML/CSS/JS files, injecting modern aesthetics.
6. **QA Node (Pro/Max)**: Launches a headless Playwright Chromium instance, loads the unpacked extension, and captures any console errors or crashes, seamlessly sending them back to the Coder for refinement.
7. **Legal & Integration**: For higher tiers, Sidekick generates a standard Terms of Service document and injects third-party APIs.
8. **Assembler Node**: Packages the finalized source code into a professional ZIP, ready for the Chrome Web Store.

## Manifest V3 (MV3) Compliance

Every extension Extensy generates follows the latest **Manifest V3 standard**. Chrome introduced MV3 to improve security and performance across all extensions. 

| Feature | Manifest V2 (Legacy) | Manifest V3 (Current) |
| :--- | :--- | :--- |
| **Background Logic** | Persistent background pages | Event-driven Service Workers |
| **Network Requests** | Full `webRequest` API | Performance-safe `declarativeNetRequest` |
| **Code Injection** | Remote code allowed | All code must be bundled in the extension |
| **Permissions** | Large, broad permissions | Granular "Least Privilege" model |

<Info>
  Extensy handles all the complexity of MV3 service workers and declarative rules for you automatically.
</Info>

## Anatomy of an Extension

When you download your ZIP, you'll generally find these files:

- **manifest.json**: The "brain" of your extension. It tells Chrome your extension's name, version, and which permissions it needs.
- **popup.html/css/js**: The small window that appears when you click the extension icon in the toolbar.
- **content.js**: Script that runs in the context of the webpages you visit. This is used for modifying DOM elements or reading page content.
- **background.js**: The service worker that listens for browser-level events like tab updates or messages from the popup.
- **icons/**: A folder containing your extension's branding in various sizes (16px, 48px, 128px).

## Subscription Tiers & Billing

Extensy uses **Polar** for secure, transparent subscription management. Depending on your tier, the backend routes to different functionality and LLMs.

- **Free Tier**: Powered by `claude-haiku-4-5`. Uses the bare-bones Coder and Assembler nodes to generate simple extensions instantly.
- **Pro Tier**: Powered by `claude-sonnet-4-5`. Activates the Architect for upfront planning, UI Designer for premium aesthetics, Playwright QA for built-in testing, and automatic Legal/TOS generation.
- **Max Tier**: Gets all Pro features plus iterative advanced research (via the Nia API) and external 3rd-party integration wiring.

## Iterative Development

The best way to use Extensy is through iteration. Start with a simple prompt, download and test the extension locally, and then come back to add advanced features or fix UI bugs.
